# 文本提取问题与后续打算

## 遇到的问题

- 当前实现把 `pdf.js` 同时用于页面渲染和文本提取。
- 在部分 PDF 里，特殊符号、emoji、图标字体的文本语义会丢失。
- 典型现象是：
  - `getTextContent()` 返回的是错误的 ASCII 标点串
  - 看起来是一个 emoji，但提取结果不是实际字符
  - 同一份文档里，有些 emoji 正常，有些不正常
- 根因不在简单编码，而在 PDF 字体语义链本身：
  - `ToUnicode` 可能缺失或无效
  - symbolic font / embedded subset font 的映射不可靠
  - `pdf.js` 能渲染这些字形，但不擅长把这类分层字形稳定还原成真实文本
- 目前已经加过一层启发式恢复，但这类方案更像 workaround，不适合作为长期主路径。

## 打算

- 不再把 `pdf.js` 作为主文本提取方案。
- 后续方向是把项目改成以 Python 为核心实现，只保留 `uvx` 作为运行入口。
- `pdf.js` 这条线如继续保留，最多只承担渲染或兼容性角色，不再负责主文本语义抽取。
- 新实现会优先选择更适合文本提取的 PDF 引擎，重点看：
  - 字体语义恢复能力
  - 对缺失 `ToUnicode` 场景的表现
  - 安装和分发是否适合 `uvx`
- 目标不是继续堆更多特例修补，而是换主链路，先把文本提取做稳，再考虑图像、OCR、回退策略。
