# 文本提取迁移方案

## 目标

这次迁移的目的不是把实现从 Node 改成 Python 本身，而是解决当前主链路在特殊字符上的不稳定问题，尤其是 emoji、symbolic font、图标字体以及 `ToUnicode` 缺失或失真的 PDF。

迁移是否成立，只看一件事：新主链路在已知坏样本上的真实文本读出率，必须明显好于当前 `pdf.js` 方案；如果做不到，就不迁。

同时保留项目当前定位：

- 继续提供开箱即用的本地 PDF MCP server。
- 继续面向 `uvx` 启动和 PyPI 分发。
- 继续保持客户端接入足够简单，迁移后仍应只需少量配置即可启动。
- 继续保留简单的 `read_pdf` 工具接口。
- 继续保留多模态输出，让模型能同时看到文本、页面结构和页面图像。

## 不做什么

以下内容不作为首版迁移方案的一部分：

- 不再继续堆字符级启发式修补。
- 不做正则替换式 workaround。
- 不引入 OCR。
- 不把“模型看图也许能读懂”当成文本失败后的成功标准。
- 不因为迁移而放弃 `Apache-2.0`。

多模态仍然重要，但它的作用是补充页面结构、图片内容和复杂版面信息，而不是替代服务端正确输出文本。

## 约束

首版迁移同时受以下约束：

- 项目继续保持 `Apache-2.0`。
- 首版以 Python 为核心实现，默认通过 `uvx` 运行。
- Codex 和其它 MCP 客户端的接入方式必须继续保持简单，目标是几行 `uvx` / JSON 配置即可运行。
- 首版优先单引擎方案，链路保持清晰。
- 不做 OCR，不做字符级修补。

这意味着候选技术栈必须同时满足：

- 许可证与 `Apache-2.0` 分发兼容。
- 能在 `uvx` 场景下尽量直接安装和运行。
- 不要求用户额外编排复杂启动脚本、桥接进程或本地服务。
- 既能支持文本提取，也能支持稳定的页面图像输出，或者至少能和渲染组件清晰解耦。

## 成功标准

迁移成功不是抽象判断，而是验收结论。

必须先建立一组固定坏样本，至少覆盖：

- emoji 读不对或读丢的 PDF
- symbolic font / 图标字体读错的 PDF
- `ToUnicode` 缺失、异常或不可靠的 PDF
- 正常数字文本 PDF
- 图文混排和复杂版面 PDF

新方案必须同时满足：

- 在坏样本上的目标字符读出率明显高于当前实现。
- 正常文本页不能出现明显退化。
- `auto` 模式仍然稳定返回页面图像。
- 启动和运行不能污染 `stdio`。
- 客户端接入体验不能明显复杂化，`uvx` 配置复杂度应与当前 `npx` 配置同级。
- Windows 作为主平台必须通过；macOS / Linux 至少完成安装与启动验证。

如果单引擎方案在这组样本上达不到门槛，结论应当是“暂不迁移”，而不是降低目标或重新引入脆弱修补。

## 新架构方向

### 运行形态

- MCP server 主实现迁移到 Python。
- 默认发布到 PyPI，通过 `uvx pagebraid-mcp` 启动。
- Codex 中应继续只需少量 `toml` 配置即可接入，其它客户端应继续只需少量 JSON 配置即可接入。
- 下一版直接去掉 Node 主线，仓库和发布物都以 Python 实现为准。

### 工具接口

首版仍然保留单工具 `read_pdf`，因为它符合当前项目“调用简单”的目标。

输入继续保持简洁：

- `file_path`
- `mode = auto | text_only | image_only`
- `pages = "23" | "23-27" | "23-"`

输出语义继续保持：

- `auto` 返回文本和页面图像
- `text_only` 只返回文本
- `image_only` 只返回图像
- 继续保留 budget-first、截断说明和 continuation 建议

但是要补一条新的质量语义：

- `text_only` 和 `auto` 中的文本块都以“尽力返回真实文本”为目标。
- 页面图像用于帮助模型理解页面结构、图片内容和难页，不用于掩盖文本主链路失败。

### 文本与渲染

不再默认把“文本提取能力”和“页面渲染能力”绑死在同一个历史实现上。

即使首版最终采用单引擎，也要在架构层面把两者拆成独立模块：

- 文本提取模块负责页级文本产出。
- 页面渲染模块负责图像产出。
- `read_pdf` 服务层负责页范围、payload budget、缓存、摘要和 continuation。

这样做不是为了提前做复杂回退，而是为了避免再次把渲染能力误当成文本语义能力。

## 实施顺序

### 第一阶段：候选评测

先不动现有对外接口，先做候选文本引擎评测。

评测只回答四个问题：

- 在坏样本上，是否明显优于当前 `pdf.js`
- 在正常样本上，是否没有明显退化
- 是否满足 `Apache-2.0` 许可约束
- 是否适合 `uvx` 的安装分发

候选评测通过前，不进入正式迁移。

### 第二阶段：Python 主线落地

在候选引擎过线后，再开始 Python MCP server 实现，优先完成：

- `read_pdf` 工具注册
- 本地 PDF 路径解析
- 页范围解析
- 文本提取
- 页面渲染
- payload budget
- 截断摘要与 continuation
- 进程内缓存
- `stdio` 安静运行

### 第三阶段：替换与收口

当 Python 版本完成且验收通过后，再处理仓库层面的收口：

- README 安装方式改为 `uvx`
- 文档更新为 Python 主线
- 旧 Node 主线降级为过渡实现或移除

## 验收结论规则

这次迁移只接受两种结论：

- 候选单引擎明显优于当前坏样本基线，进入并完成迁移。
- 候选单引擎达不到目标，停止迁移，重新规划。

不接受以下结论：

- 文本没有明显提升，但因为多模态还能看图，所以也算成功。
- 单引擎不够，但先靠启发式字符修补过渡。
- 先迁过去，之后再补 OCR 或别的兜底。

这些做法都会偏离迁移初衷。
